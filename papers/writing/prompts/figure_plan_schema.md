---
name: figure_plan_schema
description: Schema for ai/config/figure_plan.json — the authoritative registry of all figures planned, generated, and placed in the paper
---

PURPOSE:
The figure plan tells the writing system:
- what figures are planned for each section
- what type each figure is (results visualization, TikZ concept diagram, dataset sample)
- what data or code each figure draws from
- which manuscript claim each figure supports
- the current generation status

CORE PRINCIPLE:
The figure plan is the single source of truth for figure creation and placement.
No figure may be placed in a section without a corresponding entry in the plan.
No figure may be generated without an authoritative data source declared in the plan.

---

# FIGURE TYPES

## results_visualization
A multi-panel or single-panel figure generated by a Python script from experiment outputs.
Source: one or more files declared as authoritative in ai/config/results_manifest.json.
Generator: writing/prompts/figure_script_writer.md produces the script; script is run to produce the image file.
Example: prediction comparison grid, metric bar chart, ablation comparison panels.

## tikz_inline
A concept or architecture diagram written as LaTeX TikZ code directly in the section source.
No image file is generated; the code is embedded in the .tex file.
Generator: writing/prompts/figure_tikz_writer.md produces the TikZ code.
Example: model architecture diagram, pipeline flowchart, conditioning variant diagram.

## dataset_sample
A figure composed of representative images from the dataset, illustrating data characteristics.
Source: dataset image files declared in the plan.
Generator: writing/prompts/figure_script_writer.md produces a layout/composition script.
Example: sample input images with ground-truth overlays.

## manual
A figure that must be created by the author; the writing system cannot generate it.
Example: custom hand-drawn diagram, external tool export, photograph.
The plan entry must include a clear description of what is needed so the author knows what to produce.

---

# REQUIRED FIELDS PER ENTRY

```json
{
  "figure_id": "<short_slug>",
  "type": "results_visualization | tikz_inline | dataset_sample | manual",
  "section": "methodology | results | numerical_results | experiments | introduction | conclusion",
  "placement_hint": "<where in the section: after paragraph describing X>",
  "purpose": "<one sentence: what this figure shows and why it is needed>",
  "claim_link": "<the specific manuscript claim this figure supports>",
  "data_source": {
    "type": "results_manifest_group | dataset_files | none",
    "id": "<results manifest experiment_group id, or file path list, or null>"
  },
  "output_path": "ai/out/figures/<section>/<figure_id>/<figure_id>.pdf",
  "script_path": "ai/out/figures/<section>/<figure_id>/generate.py",
  "caption": "<full LaTeX caption text>",
  "label": "fig:<figure_id>",
  "status": "PLANNED | SCRIPT_WRITTEN | GENERATED | PLACED | FAILED | MANUAL_PENDING"
}
```

---

# STATUS DEFINITIONS

| Status | Meaning |
|---|---|
| PLANNED | Entry created; no script or file yet |
| SCRIPT_WRITTEN | generate.py written but not yet run |
| GENERATED | Image file exists at output_path |
| PLACED | \includegraphics{} or TikZ code emitted in section .tex |
| FAILED | Script ran but produced no valid output |
| MANUAL_PENDING | type=manual; author has not yet supplied the file |

# OPTIONAL FIELDS

```json
{
  "author_suggestion": "<verbatim text from figure_suggestions.md if this entry was author-requested>",
  "multimodel_origin": "gpt | claude | consensus",
  "rejected_alternatives": ["<description of alternative figure design considered and rejected>"]
}
```

# FILE LOCATIONS

Primary plan file:
- ai/config/figure_plan.json

Author suggestions (optional input, not modified by pipeline):
- ai/config/figure_suggestions.md

Generated scripts:
- ai/out/figures/{section}/{figure_id}/generate.py

Generated image files:
- ai/out/figures/{section}/{figure_id}/{figure_id}.pdf

---

# VALIDATION RULES

Required:
- Every entry has figure_id, type, section, purpose, claim_link, status
- results_visualization entries must have data_source.type = results_manifest_group
- results_visualization entries must reference a group with include_in_paper = yes
- tikz_inline entries must have output_path = null and script_path = null
- No two entries share the same figure_id
- No entry may reference a results_manifest group not present in ai/config/results_manifest.json

Forbidden:
- Entries without a claim_link
- results_visualization entries referencing excluded or deprecated experiment groups
- PLACED entries without an existing output_path file (for non-tikz types)
